# Issue 分发机制（PM/产评专用）

## 1. 分发目标

- 保证每个 Issue 可执行、可验收、可追踪
- 控制单人/单 Agent 负载，避免并发过载
- 保证前后端依赖顺序正确

## 2. 拆解粒度

单个 Issue 建议满足：

- 预计 0.5~2 天
- 改动 1 个主目标
- 验收标准 <= 5 条

不符合则继续拆分。

## 3. 分发规则

### 3.1 依赖优先

- 先分发上游依赖（通常 BE 契约）
- 再分发下游对接（通常 FE 集成）

### 3.2 容量上限

- 每个执行人同时 `doing` 不超过 2 个 Issue
- 阻塞超过 1 天必须升级到 PM 处理

### 3.3 验收前置

- 没有验收标准不允许下发
- 没有 Spec 链接不允许下发

## 4. 任务池结构

- Ready：可立即开始
- Blocked：依赖未满足
- Verify：开发完成待验收
- Done：通过验收

## 5. 周期内操作指令（PM）

每日最少做三件事：

1. 清理 Blocked（移除或降级）
2. 调整 Ready（补充高优先级）
3. 推进 Verify（减少“已做未验”堆积）

## 6. 升级机制

满足任一条件应升级：

- P0 任务阻塞超过 4 小时
- 关键路径任务连续 2 天无进展
- 发现跨仓库契约冲突

升级动作：

- 新建 `BLOCKER` Issue
- @PM + 架构 + 相关 FE/BE 负责人
- 24h 内给出替代路径或降级方案
