# PM / 产评经理治理手册（Issue 驱动 + AI Native）

## 1. 定位

你们在这个体系中的角色不是“写代码的人”，而是：

- 定义目标与边界
- 分配可执行任务
- 管控风险与优先级
- 用指标驱动迭代

本手册用于指导 `blog-FE` 与 `blog-BE`（含 Agent）在统一流程下推进。

## 2. 端到端流程（你们每一步该做什么）

### 阶段 A：需求定盘（PM 主导）

输入：需求想法、业务目标、时间窗口

产出：Spec 草案（使用 `templates/spec-template.md`）

必须明确：

- 业务目标（可量化）
- MVP 范围（必须做 / 不做）
- 验收口径（成功标准）

### 阶段 B：产评评审（产评经理主导）

输入：Spec 草案

产出：评审结论（通过 / 修改 / 拒绝）

评审维度：

- 可实现性：当前 FE/BE 能否支撑
- 依赖风险：是否依赖未完成模块
- 交付风险：是否超过迭代容量
- 验证风险：是否具备可验证路径

### 阶段 C：Issue 拆解与分发（PM + 产评联合）

输入：通过评审的 Spec

产出：FE/BE/Docs/Ops Issues

拆解标准：

- 每个 Issue 1~2 天可完成
- 每个 Issue 有单独验收标准
- 每个 Issue 只能有一个主负责人
- 每个 Issue 必须链接到 Spec

### 阶段 D：执行与日同步（研发/Agent执行，PM跟踪）

输入：Issue 列表

产出：每日状态变更（todo/doing/verify/done/blocked）

PM 日常动作：

- 清理阻塞
- 调整优先级
- 收敛范围蔓延

### 阶段 E：验收与发布（产评经理主导）

输入：PR 与 Harness 验证证据

产出：验收结论 + 发布决策

发布门槛：

- 关键链路 smoke 通过
- 核心指标不退化
- 已有回滚策略

### 阶段 F：复盘与回写（PM 主导）

输入：本周执行数据、缺陷、返工

产出：规则优化项、模板优化项、下周计划

## 3. Issue 分发矩阵（谁拿什么）

### FE Agent（前端）

- 负责：页面、交互、状态、前端 API 适配
- 不负责：后端数据语义定义

### BE Agent（后端）

- 负责：模型、接口、鉴权、错误语义
- 不负责：前端展示决策

### Docs Agent（文档控制面）

- 负责：Spec/流程/模板/回写
- 不负责：业务代码实现

### PM Agent（编排）

- 负责：拆解、分配、节奏、冲突协调

### QA/Verify Agent（验证）

- 负责：构建、测试、关键链路验证、回归记录

## 4. Issue 模板中的 PM 必填字段

每个 Issue 都要补充这 6 项：

1. 业务价值（为什么做）
2. 范围边界（做什么 / 不做什么）
3. 验收标准（可验证）
4. 依赖关系（依赖谁）
5. 风险等级（P0/P1/P2）
6. 截止时间（本迭代内）

## 5. 优先级规则（统一口径）

- P0：阻塞主链路（注册-登录-发帖-搜索）
- P1：影响体验/交付节奏的核心项
- P2：优化项，可延后

优先级调整原则：

- 阻塞 > 新需求
- 修复 > 新功能
- 稳定性 > 样式优化

## 6. 周节奏（建议）

- 周一：Spec 评审 + Issue 分发
- 周二~周四：执行 + 每日同步
- 周五：验收 + 发布窗口 + 复盘回写

## 7. 迭代优化闭环

每周必须回答 4 个问题：

1. 本周返工最多的原因是什么？
2. 哪类 Issue 最容易延期？
3. 哪条规则缺失导致 Agent 越界？
4. 下周要新增/修改哪条规范？

并把结论沉淀到：

- `AGENTS.md`（行为规则）
- `templates/*`（模板升级）
- 本文档（流程升级）
