# 阶段审核总流程（Stage Gate Policy）

## 目标

把审核从“主观评审”升级为“分阶段、可执行、可追踪”的治理流程。

## 阶段定义

### Stage A：MVP 基础搭建（功能先行）

目标：核心链路先打通。

合并标准：

- 核心功能可用（注册-登录-发帖-搜索）
- 不引入安全/数据破坏风险
- 有最小可验证证据（构建或基本运行）

审核策略：

- 只强制 `BLOCKER`（必须修）
- 非关键优化一律转 Follow-up Issue，不阻塞合并

### Stage B：稳步优化（质量抬升）

目标：在功能可用基础上提高稳定性和一致性。

合并标准：

- Stage A 标准全部满足
- 代码格式、提交规范达标
- 基础测试/构建通过
- 契约一致（API 结构、错误语义）

审核策略：

- `BLOCKER` + `REQUIRED` 必须修
- `FOLLOW-UP` 可延后但必须建 Issue

### Stage C：规范迭代（工程化闭环）

目标：形成长期可扩展的 AI Native 工程系统。

合并标准：

- Stage B 标准全部满足
- Harness 证据完整（执行、观察、验证、收敛）
- 经验回写到 docs/rules/templates

审核策略：

- 所有流程项可审计
- 没有回写与验证证据不合并

## 阶段切换条件

- A -> B：核心链路稳定通过 1 周，P0 缺陷清零
- B -> C：连续两个迭代保持高完成率和低返工
