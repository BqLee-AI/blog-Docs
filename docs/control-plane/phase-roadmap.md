# Phase Roadmap

## 目标

定义 Phase 0/1/2/3/4 的长期阶段边界，确保 Agent 在推进时不会把后续能力提前拉入 Phase 1。

## Phase 0：控制面建模与基线对齐

- 阶段目标：
  - 建立控制面分层职责（control-plane/runtime/harness/contracts/workflow/standards）。
  - 冻结 Phase 1 MVP 边界与验收入口。
  - 建立后续 Docs/FE/BE 任务拆分框架。
- 明确不做：
  - 不实现 FE/BE 业务能力。
  - 不进行完整 OpenAPI 大修。
- 进入下一阶段的大概条件：
  - Phase 1 范围文档稳定可引用。
  - 路线图与范围文档能够指导任务下发。

## Phase 1：MVP 最小内容闭环

- 阶段目标：
  - 邮箱验证码注册/登录。
  - JWT 鉴权与受保护写操作。
  - 登录用户完成文章创建、编辑、删除、发布。
  - 游客浏览已发布文章列表与详情。
  - 文章数据持久化到 PostgreSQL，前端基于真实 API 读取。
- 明确不做：
  - 评论、回复、点赞。
  - TinyMCE 富文本。
  - 图片上传、MinIO、OSS。
  - 全文搜索。
  - reCAPTCHA。
  - 复杂个人中心、复杂后台统计。
  - 完整 CI/CD、E2E、监控告警。
- 进入下一阶段的大概条件：
  - 认证与文章核心链路形成“可注册登录 + 可写可读 + 游客可浏览”的端到端闭环。
  - Phase 1 范围内能力不依赖 mock 才能完成主流程。

## Phase 2：内容交互与编辑体验增强

- 阶段目标：
  - 引入评论/回复/点赞等读者交互能力。
  - 引入富文本编辑能力（如 TinyMCE）并完成内容渲染一致性。
  - 引入图片上传及对象存储链路（MinIO/OSS）。
- 明确不做：
  - 不将全文搜索、复杂后台统计、完整平台级运维体系作为本阶段必达项。
- 进入下一阶段的大概条件：
  - 内容编辑和读者互动链路可用且与权限模型一致。
  - 富文本与图片链路具备稳定的数据落库与读取能力。

## Phase 3：检索、运营与管理能力扩展

- 阶段目标：
  - 建立全文搜索能力。
  - 补齐复杂个人中心和后台统计/运营视图。
  - 形成更完整的管理流程与运营分析支撑。
- 明确不做：
  - 不以完整 CI/CD、全量 E2E、全栈监控告警作为本阶段唯一出口条件。
- 进入下一阶段的大概条件：
  - 搜索与运营能力稳定服务核心业务路径。
  - 管理侧能力达到可持续运营的最小标准。

## Phase 4：工程化与可靠性收敛

- 阶段目标：
  - 建立完整 CI/CD 流程。
  - 建立分层测试体系（含 E2E 自动化）。
  - 建立监控、告警与运行保障体系。
  - 持续推进性能、稳定性与发布效率优化。
- 明确不做：
  - 不扩张新的高复杂业务域作为阶段主目标。
- 进入下一阶段的大概条件：
  - 本阶段为工程化收敛阶段，达到发布与运维可持续后进入常态化迭代。
