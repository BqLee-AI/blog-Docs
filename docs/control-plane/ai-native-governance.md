# AI Native 治理总则（blog-Docs）

## 目标

将 `blog-Docs` 建成前后端开发的控制面仓库，而不是资料仓库。

## 三层控制面

1. Prompt 控制：任务边界与验收口径
2. Context 控制：信息权威、新鲜度、加载范围
3. Harness 控制：执行、验证、回滚、审计闭环

## 治理原则

- 先定盘，后开发：Spec 和边界先于编码
- 先验证，后合并：没有验证证据不合并
- 先回写，后遗忘：每次踩坑必须沉淀为规则
- 先模块化，后扩张：规范短小、可复用、可演进

## 仓库职责边界

- 本仓库负责：规则、流程、模板、闭环机制
- FE/BE 仓库负责：业务实现与测试执行
- 本仓库不替代 FE/BE 的技术设计细节，但定义其交付约束

## PM 参与模式

传统流程里 PM 和开发是割裂的——PM 写 PRD 丢过去，开发写代码丢回来，中间全靠口头对齐。AI Native 流程改变了这个关系：

### 核心变化

1. **PM 通过 GitHub Discussions + OpenSpec 直接参与仓库治理**，不是外围旁观
2. **blog-Docs 的 OpenSpec 不写代码，只冻结决策**——立什么 Issue、不做什么、验收标准是什么
3. **这些决策是传统开发者最容易模糊的地方**——他们擅长写代码，不擅长收边界

### 传统 vs AI Native

| 传统流程 | AI Native 流程 |
|---------|---------------|
| PM 写 PRD → 丢给开发 → 开发自己理解 → 做完再说 | PM 在 Discussions 讨论 → OpenSpec 冻结决策 → Issue 精准下发 |
| PM 只能看最终效果 | PM 在每一步都能参与：spec 审核、PR 验收、回写同步 |
| 开发者自己猜边界 | Issue 里已经写了非目标、实现约束、输入真源 |
| 验收靠感觉 | 验收对照 spec scenario 逐条检查 |
| 经验留在人脑 | 经验沉淀为规则、回写到文档仓库 |

### 控制面职责

文档仓库的核心职责是**把模糊变成冻结**：

```
模糊讨论（Discussions）
    → /opsx-explore 想清楚
    → /opsx-propose 冻结成 proposal/design/specs/tasks
    → 从 tasks 拆 Issue 下发到 FE/BE
    → 开发者按冻结的 spec 执行
    → PM 对照冻结的 spec 验收
    → /opsx-archive 归档 + 回写
```

文档仓库不是在替开发写代码，而是在替所有人把决策外置成可追溯的资产。

## 运行节奏

- 本仓库只沉淀长期有效的规则、模板、边界与闭环机制
- 周期性状态、周计划、候选版本进度应存放在外部同步记录，而不是写入治理总则
- 开发范式可以演进，但应通过修订规则与模板体现，而不是在总则里保留阶段口号
