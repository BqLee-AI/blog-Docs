# 文档落点规范

## 目的

明确一条信息应该写到哪里，避免控制面变成资料堆，也避免 Agent 因为找不到真源而重复写规则。

## 落点总则

- 高频硬规则放 `AGENTS.md`。
- 长期范围和治理边界放 `docs/control-plane/`。
- 流程步骤和协作协议放 `docs/workflow/`。
- 验收口径和证据要求放 `docs/harness/`。
- 可复用标准放 `docs/standards/`。
- 当前阶段状态、Issue 派发、风险和差异台账放 `docs/runtime/`。
- 可直接复制使用的表单放 `templates/`。
- API 契约真源放 `contracts/`。
- 待确认的治理变更放 `openspec/changes/<change-id>/`。

## `AGENTS.md` 放什么

适合放：

- Agent 开工必须遵守的读取顺序。
- 默认语言和标题风格。
- OpenSpec 触发条件。
- 原子 Issue / PR 规则。
- 验收证据和回写要求。
- 禁止把临时状态写进固定层。

不适合放：

- 大段解释、示例和历史背景。
- 阶段任务明细。
- 某个 Issue 或 PR 的当前状态。
- templates 的完整字段说明。

## `docs/control-plane/` 放什么

适合放：

- MVP scope。
- phase roadmap。
- 控制面职责边界。
- 长期有效的红线和高权重真源定义。

不适合放：

- 每个 Issue 的当前进度。
- 某个 PR 是否已合并。
- 临时风险排查过程。

## `docs/runtime/` 放什么

适合放：

- 当前阶段状态。
- Issue 派发表。
- 风险登记。
- OpenAPI gap 当前清单。
- 阶段回写记录。

不适合放：

- 永久规则。
- 命名规范。
- 模板写法。
- MVP 边界原则。

## `docs/standards/` 放什么

适合放：

- Agent 工作规范。
- 命名规范。
- 文档落点规范。
- 模板编写规范。
- AGENTS.md 编写标准。
- Spec 写作标准。
- FE/BE 接入门禁。

不适合放：

- 某个阶段的当前任务列表。
- 一次性执行记录。
- 未经确认的想法。

## `templates/` 放什么

适合放：

- Issue 模板。
- PR 模板。
- Spec 模板。
- Harness 清单。
- 可复制的执行表单。

不适合放：

- 长篇规则解释。
- 历史示例合集。
- 不能直接填写的概念说明。

## 不需要记录的内容

以下内容默认不进入文档仓库：

- 临时聊天过程。
- 没有形成决策的想法。
- 已经体现在 git diff 中的琐碎修改说明。
- 一次性命令输出，除非它是验收证据。
- 已关闭且无复用价值的中间方案。
- 重复描述 OpenAPI 或代码中已经存在的细节。

## 验收检查

- 这条信息是否会长期复用？
- 它是规则、状态、模板、契约还是证据？
- 它是否已有更高权重真源？
- 它是否会很快过期？如果会，应进入 runtime 或不记录。
- 它是否改变 scope、contract、harness 或 gate？如果改变，应先走 OpenSpec。
