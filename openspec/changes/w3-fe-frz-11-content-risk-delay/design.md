## Overview

本次变更的设计目标不是把 Week 3 的链路直接推进实现，而是先把候选版本边界、验证口径与回滚路径前置到 OpenSpec change 中，保证发布壳只吸收已冻结范围内的内容。

## Design Decisions

### 1. 冻结边界优先于实现推进

- FE-009 / FE-010 / FE-011 / FE-012 相关内容先进入 OpenSpec 前置，不直接进入实现分支。
- 候选版本的范围判断优先依据冻结边界，不以临时开发进度判断是否可合并。

### 2. 短期状态外置

- 候选版本状态、风险台账、烟 smoke 记录放在低权重同步记录中。
- 稳定规则正文只承载长期有效的边界、门禁和回滚要求。

### 3. Harness 计划先行

- 在允许进入实现前，先明确构建、关键链路 smoke、回滚口径与证据格式。
- 若缺失验证口径，则保持冻结状态，不放行进入候选版本。

## Artifact Mapping

- `proposal.md`: 说明为什么要先冻结、为什么要延期。
- `specs/fe-release-freeze-boundary/spec.md`: 定义候选版本冻结边界、延期准则与验证要求。
- `tasks.md`: 定义需要完成的同步记录、验证计划和回写动作。

## Failure / Rollback Path

- 若发现 FE-009 / FE-012 仍需补充契约或验证前置，则继续保持冻结，不扩展候选版本。
- 若冻结边界被证明过严，可通过新的 OpenSpec change 调整范围，但不得直接在实现层松绑。
